# 常见问题

## 1. 为什么不直接把所有文档发给大模型？

因为文档可能太长，会超过上下文长度；而且成本高、速度慢。检索可以先筛出最相关的片段。

## 2. 为什么要切块？

切块能让检索更精确。用户问一个具体问题时，系统只需要找到相关段落，不必处理整篇文档。

## 3. 切块大小怎么选？

新手可以先用 300 到 800 个字符。资料越复杂，片段可以稍微长一点；问答越精确，片段可以稍微短一点。

## 4. overlap 是什么？

overlap 是相邻切块之间的重叠部分。它可以降低“重要句子被切断”的风险。

## 5. embedding 模型和聊天模型有什么区别？

embedding 模型负责把文本变成向量，方便检索和聚类。

聊天模型负责生成自然语言回答。

## 6. 为什么检索出来的片段有时不准？

常见原因有：

- 文档切块太粗或太碎。
- embedding 模型不适合当前语言或领域。
- 用户问题太短，语义信息不足。
- 没有 reranker 做二次排序。

## 7. RAG 一定不会胡编吗？

不会。RAG 只能降低胡编概率，不能彻底消除。更严谨的系统需要显示引用来源，并要求答案必须基于检索片段。

## 8. 为什么项目里有 hashing 后端？

hashing 后端是教学和环境兜底方案。它不是真正的 Transformer，但能帮助你先跑通“切块、建索引、检索、回答”的工程流程。

真正学习 Transformer 时，重点看 `TransformerEmbedder`。

## 9. 为什么要指定模型缓存目录？

Transformer 模型文件可能很大。Hugging Face 默认缓存位置通常在用户目录，也就是很多 Windows 电脑的 C 盘。

为了避免占用 C 盘，本项目运行 Transformer 后端时建议显式传入：

```powershell
--cache-dir 'D:\hf-cache'
```

新手要记住：深度学习项目不仅有代码，还有模型权重、数据集、日志、缓存，这些文件最好提前规划存放位置。
